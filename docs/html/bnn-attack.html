The Bayes‑NN attack
— why huge noise doesn't save you

A token is turned into an embedding, then differentially-private Gaussian noise is added before it leaves the trusted boundary. An adversary who knows the embedding table matches the noisy vector back to its nearest table entry. Despite noise larger than the embedding itself, this recovers the token almost perfectly. This page shows, interactively, why.

§01

What the attack is

setup · one token

Everything happens at the very first layer — the embedding lookup — before any transformer block. The secret is a single token v.

  1. Embed. The model maps token v to a fixed vector e_v ∈ ℝd (d=2304) by table lookup. The table {e_u} of all tokens is public (open weights).
  2. Clip + add DP noise. The vector is clipped to norm C≈4.15 and Gaussian noise is added: Y = clip(e_v) + ε,  ε ∼ 𝒩(0, σ² I_d). Smaller privacy budget εdp → larger σ.
  3. Leak. Only the noisy Y crosses the boundary to the untrusted side.
  4. Attack (BNN). The adversary returns the table entry closest to Y: v̂ = argminu ∥Y − e_u∥. No training, no model — just nearest neighbour against the known table. This is the Bayes-optimal (MAP) decoder under Gaussian noise.

The puzzle: at the privacy level εdp=64 the noise vector has length ∥ε∥≈15 — about 3.6× the embedding's own length (≈4.15). Intuitively the signal is buried. Yet BNN recovers the token ~94% of the time. The next section shows why that intuition is wrong.

§02

Why huge noise doesn't break it

interactive · the key idea

BNN does not need Y to land near e_v. It only needs e_v to be the closest of all candidates. That is a question about one direction — the axis between two codewords — not about the whole noise vector.

Compare the true token e_v to one rival e_u. Split the noise ε into two parts: its component along the axis Δ = e_v − e_u (this is all that decides v vs u), and everything perpendicular to it (the other d−1 directions). The perpendicular part moves Y far away — but it is the same distance to e_v and to e_u, so it cancels in the comparison. The decision sees only the along-axis noise, whose size is just σ — one coordinate's worth.

Slide d up: the total noise ∥ε∥≈σ√d balloons (watch the total noise readout and the ⊥-junk note), but the bumps, the decision SNR, and the error do not move at all. That is the whole resolution: extra dimensions only add perpendicular junk that cancels. Slide εdp down (more noise): the along-axis spread σ grows relative to the gap, the two bumps widen and overlap, the SNR falls, and dots begin crossing the boundary. Switch to a morphological twin (tiny gap): the bumps sit almost on top of each other and merge into a coin flip even at modest noise — those rare close pairs are the only place BNN fails (§04).

§03

The decision, in one number

SNR → error

Formally, the margin "is e_v closer than e_u?" is ∥Y−e_u∥² − ∥Y−e_v∥² = ∥Δ∥² + 2⟨ε,Δ⟩. The random part 2⟨ε,Δ⟩ has standard deviation 2σ∥Δ∥, so the probability BNN picks the wrong one is exactly Φ(−∥Δ∥/2σ) — set entirely by the decision SNR ∥Δ∥/2σ. The full noise norm ∥ε∥ never appears.

That single formula also says how to actually defeat the attack: shrink the SNR by pushing εdp down into the real DP regime (εdp≈0.1–10). Slide the budget left (in Fig A or below) and watch what happens — and which pairs blur first.

Figure B — per-pair error vs privacy budget (log ε)synced to Fig A εdp

Per-rival error as εdp sweeps from 256 (no privacy) down to 0.1 (strong DP), log scale. 0.5 = a coin flip = the two words are indistinguishable. Note the order: the morphological twin (gap 0.70) reaches the coin flip first — DP erases the distinction between similar words while still leaving typical words recoverable (the deniability band). Only at εdp≦1 does even the typical rival (gap 5.86) blur — by then every word is gone. The marker tracks Fig A.

The catch (the utility–privacy cliff). DP does stop BNN — but look at the cost. The same noise that lifts the curve also corrupts the embedding the downstream model needs. The leaked signal I(V;Y) = log₂K − H(V|Y) is simultaneously what the attacker reads and what the model receives — the same bits. To send the twin-error to a coin flip you must reach εdp≈10; to blur typical words you need εdp≦1, where the vector is pure noise and accuracy collapses. So "noise it and ship it" buys confidentiality only by destroying utility. (For genuine token confidentiality against an untrusted cloud, the fix is not bigger noise but a different mechanism — TEE / masking / crypto.)
§04

From one rival to 2048 — where errors actually come from

the union bound · the floor

A token has 2047 rivals, not one. BNN errs if any rival wins, so the total error is at most the sum of the per-rival errors (the union bound). But that sum is dominated by the very few close rivals — almost every pair is near-orthogonal (gap ≈5.86, error ≈0). So a token's fate is decided by its single nearest neighbour.

These nearest neighbours are not random. The very closest few are rare/under-trained glyph tokens (private-use and exotic Unicode) that sit almost on top of each other in the tail of the table; the recognizable floor — the human-legible part of the list — is morphological / case / number twins. Below are those legible closest pairs from the gemma-2-2b pool, with the per-pair error each contributes at εdp=16 (highest noise tested). Either way the point holds: the geometry alone — no attack run — already names which tokens will be confused.

§05

What our probes measure — and why they explain this

bits & bounds

The takeaway. At L0 the leakage is the embedding table's distance geometry. High dimension makes the codewords well-separated and the DP noise mostly perpendicular junk, so nearest-neighbour matching survives noise far larger than the signal. H(V|Y) measures the residual matching ambiguity in bits; the union bound names the few confusable pairs. Together they predict the attack from geometry alone — which is exactly what a faithful leakage probe should do.
