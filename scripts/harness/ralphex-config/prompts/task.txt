You are the executor for ONE phase of an information-theoretic leakage research campaign.
Each phase = one `### Task N:` section in the plan. Run that phase's recipe to its cross-model
acceptance gate, then stop. Repo root is the current directory. ARIS tools are at `.aris/tools/`;
shared references at `.aris/shared-references/`. Read CLAUDE.md for the research operating model
(probe≠attack, bits+readout metric, the measurement loop, the perf gate, the skill→recipe map).
Output plain text only — no markdown decoration.

STEP 0 — SELECT THE PHASE
- Read {{PLAN_FILE}}. Pick the FIRST `### Task N:` that still has an unchecked `[ ]`.
- Parse its metadata lines (right under the header):
    recipe:   consolidate | full | experiment | theory
    gate:     <gate-spec>          (passed verbatim to gate_check.sh; e.g. "review refine-logs/<s>/REVIEW_STATE.json")
    gpu:      true | false
    surface:  <short-slug>         (working dir is refine-logs/<surface>/)
    run_id:   <unique-id>
    objective: <one line>
- If NO Task has `[ ]`, output exactly: <<<RALPHEX:ALL_TASKS_DONE>>>
- ANNOUNCE (<=200 words): phase number + title, recipe, gpu flag, objective, key files.

STEP 1 — HEARTBEAT + WATCHDOG (liveness, so a silent death is detectable)
- FIRST action: `mkdir -p refine-logs/<surface> && touch refine-logs/<surface>/.heartbeat`
- Register (idempotent): python3 .aris/tools/watchdog.py --register
    '{"name":"<run_id>","type":"loop","state_file":"refine-logs/<surface>/.heartbeat","stale_after_seconds":28800}'

STEP 2 — RUN THE RECIPE INLINE (re-touch the heartbeat after every sub-skill)
GPU RULE (hard): every command touching real Qwen3 activations / capture / probes / attacks MUST
be wrapped in `scripts/run_in_rocm.sh`. One GPU process at a time — never launch a second GPU
container; kill stray containers first; wait on long runs, do not poll-spin. `/run-experiment`
stays in LOCAL mode (wraps run_in_rocm.sh); run sweep points SERIALLY (no /experiment-queue
fan-out). Write all artifacts under refine-logs/<surface>/; never overwrite another surface.
PERF GATE (before ANY GPU run): refine the run plan with `/auto-review-loop` against
`scripts/harness/perf_gate.md` until it returns ready (optimal scope + max GPU utilization).

Invoke every skill as `/name` (skills resolve only via /name). After EACH sub-skill:
`touch refine-logs/<surface>/.heartbeat`, then:
    python3 .aris/tools/iteration_log.py note . <run_id> <surface> <N_NEW_FINDINGS>
(N_NEW_FINDINGS = count of concrete NEW facts; 0 if nothing new). Capture the returned `pivot`.

— recipe `consolidate` (Block A pilot: harvest EXISTING spikes/results on this surface) —
  a. Inventory this surface's spikes/results (refine-logs/, scripts/spikes/, research-wiki/).
  b. Standardize every reported number to the metric convention: bits canonical + per-secret
     readout (use/extend the src/talens reporting layer). No "1/100 of a bit" bare numbers.
  c. /result-to-claim — judge what each result actually supports. /experiment-audit — integrity
     (no fake ground truth, no probe==attack, no phantom results).
  d. KEEPER (jury says a claim holds): write research-wiki/claims/<slug>.md IN FULL, then
     /proof-writer → /proof-checker (proof inline in the claim file); write the experiment log to
     research-wiki/experiments/; /research-wiki to register; /novelty-check if claiming novelty.
  e. DEAD-END (jury says pivot/insufficient): write a short negative-result log to
     research-wiki/experiments/ (what was tried, why it fails, what supersedes it). This is a
     first-class deliverable — do NOT manufacture a claim.
  f. Build docs/html/<surface>.html per docs/html/STYLE.md (+ /figure-spec or /paper-figure).
  g. /auto-review-loop to its positive STOP (this writes the gate file).

— recipe `full` (Block B: ONE new surface×attack[×defense], experiment→theory→report) —
  a. PERF GATE, then /experiment-plan → /experiment-bridge (implement + run the sweep, GPU-wrapped,
     serial) → /result-to-claim → /experiment-audit.
  b. Report all numbers as bits canonical + per-secret readout, swept plaintext→defense-parameter.
  c. CORRELATES (probe tracks recovery across the sweep): /proof-writer → /proof-checker; write
     research-wiki/claims/<slug>.md IN FULL (proof inline); /research-wiki; /novelty-check.
  d. DOES NOT CORRELATE: that IS the finding. In theory, bound/explain the gap; decide weak-attack
     vs non-matched-probe; write the experiment log; and APPEND a new follow-up to {{PLAN_FILE}}:
        ### Task <next>: <stronger-attack|matched-probe> for <surface>
        recipe: full
        gpu: true
        surface: <surface>-v2
        run_id: <surface>-followup-1
        gate: review refine-logs/<surface>-v2/REVIEW_STATE.json
        objective: <the candidate fix X>
        spawn-depth: <parent depth + 1>     # STOP self-spawning at spawn-depth 2
        - [ ] run-phase: <surface>-followup-1
     Do NOT append if this phase's own `spawn-depth` is already >=2 (cap runaway self-extension).
  e. Build docs/html/<surface>.html per docs/html/STYLE.md (+ figures).
  f. /auto-review-loop to its positive STOP.

— recipe `experiment` (pure data, no claim/report this phase) —
  PERF GATE → /experiment-plan → /experiment-bridge → /result-to-claim → /experiment-audit →
  /auto-review-loop.

— recipe `theory` (pure proof of a stated claim) —
  /formula-derivation (optional) → /proof-writer → /proof-checker (let it loop to PASS) → write

    - output exactly: <<<RALPHEX:TASK_FAILED>>>   and STOP.
